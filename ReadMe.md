# SOCRadar ↔ Jira Integration (Forge App)

A bidirectional integration between SOCRadar and Jira, built as an Atlassian Forge app. Automatically creates Jira issues from SOCRadar security alarms and syncs changes back to SOCRadar when issues are updated in Jira.

## Features

### SOCRadar → Jira (Alarm Sync)
- Fetches new security alarms from SOCRadar via multi-tenant API
- Creates Jira issues with alarm details, severity, recommended response, and SOCRadar link
- Maps SOCRadar severity to Jira priority (CRITICAL → Highest, HIGH → High, etc.)
- Adds `socradar` and `security-alert` labels to created issues
- Supports manual sync ("Synchronize Now" button) and automatic hourly sync
- Pagination support: fetches up to 100 alarms per page across all pages
- Batch processing for automatic sync: max 50 issues per cycle to avoid Forge timeout
- Tracks last processed alarm ID to avoid duplicates

### Jira → SOCRadar (Bidirectional Sync)
- **Status sync**: Closing/resolving a Jira issue updates the SOCRadar alarm status
- **Severity sync**: Changing Jira issue priority updates SOCRadar alarm severity
- **Assignee sync**: Assigning a Jira issue updates the SOCRadar alarm assignee and adds a comment
- Loop prevention: changes originating from SOCRadar are not synced back

### Admin UI (Custom UI)
- Configuration tab: SOCRadar API settings, Jira project settings, integration behavior toggles
- Synchronization tab: Manual sync, dry run preview, alarm stats
- History tab: Sync logs with timestamps, alarm counts, and bidirectional sync events

## How It Works

### 1. Configuration
The admin opens the Forge app in Jira (Apps → SOCRadar Integration Settings) and configures:

| Setting | Description |
|---------|-------------|
| Multi-Tenant ID | SOCRadar tenant identifier |
| API Key | SOCRadar API authentication key |
| Base URL | `https://platform.socradar.com` (production) or `https://preprod.socradar.com` |
| SOCRadar User Email | Valid SOCRadar platform user email (required for bidirectional sync) |
| Project | Jira project where issues will be created |
| Issue Type | Task, Bug, Story, Epic, or Incident |
| Assignee | Optional default assignee (Atlassian Account ID) |
| SOCRadar source display | Toggle: show "SOCRadar" branding and alarm link in issues |
| Automatic Synchronization | Toggle: enable/disable hourly auto-sync |

All settings are stored in Forge encrypted secret storage (`storage.setSecret`).

### 2. Alarm Fetching (SOCRadar → Jira)

**Manual Sync:**
1. Admin clicks "Synchronize Now" in the Synchronization tab
2. App calls SOCRadar Multi-Tenant Incidents API: `GET /api/v1/multi-tenant/{tenantId}/incidents`
3. Paginates through all results (100 per page) to find alarms with `alarm_id > lastAlarmId`
4. Creates a Jira issue for each new alarm
5. Stores SOCRadar metadata as a Jira issue property (`socradar-metadata`)
6. Updates `lastAlarmId` in Forge storage

**Automatic Sync (Scheduled Trigger):**
1. Forge scheduled trigger fires every hour
2. Checks if `autoSync` is enabled in config
3. If enabled, runs the same sync logic but in **batch mode** (max 50 issues per run)
4. If there are more than 50 new alarms, remaining alarms are picked up in the next hourly cycle

**Dry Run:**
- Fetches alarms and shows preview without creating any issues

### 3. Bidirectional Sync (Jira → SOCRadar)

When a Jira issue is updated, the `avi:jira:updated:issue` trigger fires:

1. `onIssueChanged` handler reads the issue's `socradar-metadata` property
2. If the issue was not created by SOCRadar (no metadata), it is skipped
3. If the last update was from SOCRadar (`lastUpdatedBy: 'socradar'`), the flag is cleared and the event is skipped (loop prevention)
4. Changelog items are processed:

| Jira Change | SOCRadar Action | API Endpoint |
|-------------|-----------------|--------------|
| Status → Done/Closed/Resolved | Status → RESOLVED (2) | `POST /api/company/{id}/alarms/status/change` |
| Status → In Progress | Status → INVESTIGATING (1) | Same |
| Status → To Do/Open/Backlog | Status → OPEN (0) | Same |
| Status → Won't Do/Declined | Status → FALSE_POSITIVE (9) | Same |
| Status → Duplicate | Status → DUPLICATE (10) | Same |
| Priority → Highest | Severity → CRITICAL | `POST /api/company/{id}/alarm/severity` |
| Priority → High | Severity → HIGH | Same |
| Priority → Medium | Severity → MEDIUM | Same |
| Priority → Low/Lowest | Severity → LOW | Same |
| Assignee changed | Assignee updated + comment | `POST /api/company/{id}/alarm/{alarmId}/assignee` |

### 4. Loop Prevention

```
SOCRadar alarm created → Jira issue created (lastUpdatedBy: 'socradar')
                                    ↓
                    First Jira change detected
                                    ↓
                    lastUpdatedBy === 'socradar' → SKIP, clear flag
                                    ↓
                    Subsequent Jira changes
                                    ↓
                    Sync to SOCRadar (lastUpdatedBy: 'jira')
```

## API Endpoints Used

### SOCRadar Multi-Tenant API
| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/v1/multi-tenant/{tenantId}/incidents` | Fetch alarms |

### SOCRadar Incident Management API
| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/api/company/{companyId}/alarms/status/change` | Change alarm status |
| POST | `/api/company/{companyId}/alarm/severity` | Change alarm severity |
| POST | `/api/company/{companyId}/alarm/add/comment/v2` | Add comment to alarm |
| POST | `/api/company/{companyId}/alarm/{alarmId}/assignee` | Change alarm assignee |

All SOCRadar API calls use the `API-KEY` header for authentication.

**Important:** SOCRadar API returns HTTP 200 even on errors. The app checks the `is_success` field in the response body to detect failures.

### Jira API (via Forge)
| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/rest/api/3/issue` | Create issue |
| GET | `/rest/api/3/issue/{key}/properties/{key}` | Read issue property |
| PUT | `/rest/api/3/issue/{key}/properties/{key}` | Write issue property |
| GET | `/rest/api/3/user?accountId={id}` | Get user display name |
| GET | `/rest/api/3/project/search` | List projects for dropdown |
| GET | `/rest/api/3/mypermissions` | Verify admin permissions |

## Deployment

### Prerequisites
- Node.js 18+
- Atlassian Forge CLI (`npm install -g @forge/cli`)
- Forge CLI logged in (`forge login`)

### Build & Deploy

```bash
# Install backend dependencies
cd forge-app
npm install

# Install and build frontend
cd static/frontend
npm install
npm run build
cd ../..

# Deploy to Atlassian cloud
forge deploy

# Install/upgrade on your Jira site (first time or after permission changes)
forge install --upgrade
```

### Development (Tunnel Mode)

```bash
# Run with live reload for development
forge tunnel
```

### Updating an Existing Installation

```bash
# Increment version in package.json
npm version patch --no-git-tag-version

# Build frontend
cd static/frontend && npm run build && cd ../..

# Deploy
forge deploy
forge install --upgrade
```

## Configuration Details

### SOCRadar User Email
Required for bidirectional sync. Must be a valid email address registered as a user on the SOCRadar platform. This email is used for:
- Adding comments to SOCRadar alarms (requires `user_email` parameter)
- Updating alarm assignees (SOCRadar requires valid platform user emails)

Jira user emails are not accessible via the Forge API due to Atlassian privacy policies, so this config field is used as a proxy.

### Automatic Synchronization
When enabled, the scheduled trigger runs every hour and:
1. Checks for new alarms since the last `lastAlarmId`
2. Creates up to 50 Jira issues per cycle (batch limit to avoid Forge 25-second timeout)
3. If more than 50 new alarms exist, the remaining are processed in the next cycle

When disabled, the scheduled trigger still fires hourly but exits immediately without making any API calls.

### Base URL
- **Production:** `https://platform.socradar.com`
- **Pre-production:** `https://preprod.socradar.com`

Do not include `/api` at the end — the app constructs full API paths automatically.

## Troubleshooting

### 404 Error on Sync
Check that the Base URL in configuration is exactly `https://platform.socradar.com` without trailing slashes or `/api`.

### SOCRadar API Returns 200 but Fails
SOCRadar wraps errors in HTTP 200 responses with `is_success: false`. Check Forge logs for `[STATUS]`, `[SEVERITY]`, `[COMMENT]`, or `[ASSIGNEE]` error messages.

### Comment API Fails with Email Error
Ensure the SOCRadar User Email in configuration is a valid platform user email (e.g., `user@socradar.io`). Generic emails like `jira-integration@system` are rejected.

### Bidirectional Sync Not Working
1. Verify the issue has `socradar-metadata` property (created by the sync process)
2. Check that the `trigger` module is defined in `manifest.yml`
3. After deploying, run `forge install --upgrade` to register new permissions

### Version Number Error on Deploy
Increment the version in `package.json` before deploying:
```bash
npm version patch --no-git-tag-version
forge deploy
```

## Security

- All configuration (API keys, tenant IDs) stored in Forge encrypted secret storage
- Admin-only resolvers verify Jira admin permissions before executing
- SOCRadar API key is masked in the UI (only last 4 characters shown)
- No hardcoded secrets in source code
- External network access limited to declared domains in manifest
